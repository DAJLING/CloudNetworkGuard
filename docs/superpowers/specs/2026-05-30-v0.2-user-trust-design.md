# v0.2 User Trust Enhancements Design

## Goal

Make CloudNetworkGuard easier to trust, understand, and recover from. Version 0.2 should not add more risk checks first; it should help users understand why traffic is blocked, what to do next, and how to safely restore their network if anything goes wrong.

## Background

The current app already has the core guard architecture:

- Local proxy for guarded targets.
- Windows firewall and hosts fallback for direct clients.
- Static residential IP preflight.
- Exit IP binding by hash.
- Free IP provider scoring.
- DNS, TCP, TLS, and Claude web probes.
- Browser/system environment checks.
- Minimal local logs with masked IP data.
- Editable target rules through `target-rules.json`.

The main gap is not detection depth. The main gap is user confidence. A user can be blocked, but the product does not yet consistently explain the cause, recommend a safe fix, or provide an obvious emergency recovery path.

## Product Principles

1. Safety first: users must always have a clear way to restore system proxy, firewall, and hosts changes.
2. Explain before acting: every block should map to a plain-language reason and a next action.
3. Privacy by default: do not show or export full IP addresses unless the user explicitly chooses to reveal them later.
4. Make advanced concepts approachable: keep technical detail available, but do not make enum names or raw JSON the main product surface.
5. Fail closed, recover clearly: strict blocking is acceptable only if recovery and explanation are obvious.

## Target Users

- Users of Claude or Anthropic API who want to avoid risky network egress.
- Users who know they should use a stable residential network but do not necessarily understand DNS, TLS, ASN, or firewall behavior.
- Operators who need a local diagnostic report when access fails.

## Scope

### In Scope

- First-run setup wizard.
- Block reason explanation and repair guidance.
- Emergency network recovery.
- Exit IP binding management.
- Detection report page.

### Out of Scope

- Proxy or VPN purchase, switching, or account management.
- Paid IP intelligence provider integration.
- Cloud accounts, cloud sync, or remote policy management.
- Full macOS Network Extension or privileged packet filter implementation.
- Server-side account risk prediction.

## Feature 1: First-Run Setup Wizard

### User Problem

A new user needs to understand what the app protects and configure static residential IP behavior before enabling the guard. Today the user can be dropped into advanced controls without enough framing.

### Proposed Experience

Show a setup wizard when the app has no completed setup marker. The wizard should have three steps:

1. Protection scope
   - Explain that the app guards Claude and Anthropic API target traffic.
   - Explain that it may change local system proxy, firewall, and hosts entries while enabled.

2. Current network check
   - Run a check or reuse the most recent check.
   - Show masked exit IP, region, IP type, risk score, and whether provider data is available.

3. Static residential IP strategy
   - Bind current exit IP as the expected static residential IP.
   - Manually enter a static residential IPv4 address.
   - Explicitly skip only the static IP check by saving `0.0.0.0`.

### Acceptance Criteria

- A first-time user can complete setup without editing JSON.
- Skipping static IP validation requires an explicit user action.
- The wizard explains system-level changes before enabling the guard.
- The setup state is persisted locally.
- The wizard can be reopened from settings.

## Feature 2: Block Reason Explanations And Repair Guidance

### User Problem

When the app blocks traffic, users need to know why and what to do next. Raw reasons such as `STATIC_RESIDENTIAL_IP_MISMATCH` are useful internally but not sufficient as product copy.

### Proposed Experience

Create a reason catalog that maps each `CheckReason` to:

- Title.
- Plain-language explanation.
- Severity.
- Recommended actions.
- Optional primary action button.
- Optional secondary action button.

Example mappings:

### `STATIC_RESIDENTIAL_IP_REQUIRED`

- Title: Static residential IP is required.
- Explanation: The guard needs a known stable exit IP before it can safely release guarded traffic.
- Primary action: Configure static IP.
- Secondary action: Skip this check.

### `STATIC_RESIDENTIAL_IP_MISMATCH`

- Title: Current exit IP does not match the configured static IP.
- Explanation: Your network appears to be using a different exit IP than the one saved in the guard configuration.
- Primary action: Update static IP.
- Secondary action: Check current network.

### `IP_BINDING_MISMATCH`

- Title: Exit IP changed after binding.
- Explanation: The guard previously bound this device to a different exit IP fingerprint.
- Primary action: Review binding.
- Secondary action: Rebind to current exit.

### `PROVIDER_UNAVAILABLE`

- Title: IP intelligence sources are unavailable.
- Explanation: The guard cannot confirm the current IP risk because all configured provider checks failed.
- Primary action: Retry check.
- Secondary action: View provider details.

### `ENVIRONMENT_MISMATCH`

- Title: Browser or system environment looks inconsistent.
- Explanation: Time zone, language, or WebRTC local IP exposure may not match the expected network posture.
- Primary action: View environment details.

### Acceptance Criteria

- Every `CheckReason` has a catalog entry.
- The overview page shows the highest-priority explanation when blocked.
- The check matrix can show short helper text for each failed row.
- The UI does not rely on raw enum names for primary user guidance.
- Tests verify that all known reasons have explanation metadata.

## Feature 3: Emergency Network Recovery

### User Problem

If system proxy, firewall, or hosts changes break connectivity, the user needs a single obvious recovery action.

### Proposed Experience

Add a prominent recovery control labeled `Restore Network` or `Emergency Restore`. It should be accessible from the overview and settings pages.

When clicked, the app should:

1. Disable guard state.
2. Disable the system proxy.
3. Clear firewall rules created by this app.
4. Remove hosts entries between this app's markers.
5. Refresh status.
6. Append a local recovery log entry.
7. Report any partial failure clearly.

### Acceptance Criteria

- The action never removes firewall or hosts entries that were not created by this app.
- The result distinguishes full success from partial recovery.
- Partial recovery shows exact failed cleanup layer: proxy, firewall, or hosts.
- The action is safe to run repeatedly.
- Tests cover successful recovery and partial cleanup failure.

## Feature 4: Exit IP Binding Management

### User Problem

The app binds the first detected exit IP by hash and blocks later mismatches, but users do not have a clear way to inspect or manage that binding.

### Proposed Experience

Add a binding management section showing:

- Whether an exit IP fingerprint is currently bound.
- Masked current IP from the latest check.
- Last check time.
- Whether the latest check matches the binding.
- Actions to reset or rebind.

Available actions:

- Reset binding.
- Rebind to current exit IP.
- Run check again.

Destructive or security-sensitive actions should require confirmation.

### Acceptance Criteria

- Users can resolve `IP_BINDING_MISMATCH` from the UI.
- Reset and rebind actions are logged locally.
- Full IP addresses remain hidden.
- Rebinding requires a current provider IP result.
- Tests cover reset, rebind, and blocked rebind when no current IP exists.

## Feature 5: Detection Report Page

### User Problem

The overview page is useful for quick status, but diagnostics require deeper visibility into what failed and why.

### Proposed Experience

Add a detection report page with these sections:

- Overall verdict and checked time.
- Exit IP summary: masked IP, region, ASN, IP type, risk score, confidence.
- Provider details: source, error, country, region, ASN, risk score, confidence.
- Target connectivity: DNS, TCP, TLS per health host.
- Claude web probe status.
- Environment consistency: time zone, language, WebRTC local IP exposure count.
- Static IP strategy and current result.
- Exit binding status.
- Firewall and hosts status.
- Usage-rate status.

Add actions:

- Copy diagnostic report.
- Export JSON report.
- Retry check.

### Acceptance Criteria

- Reports hide full IP values by default.
- Copied reports are useful for support without exposing full IP addresses.
- Failed checks include details from the failing layer.
- The report can be generated from the latest stored check without rerunning network calls.
- Tests cover report generation and IP masking behavior.

## Suggested Information Architecture

The app should evolve toward these primary areas:

- Overview: current verdict, protected traffic status, top risk, primary action.
- Fixes: current block reason, explanation, and recommended actions.
- Report: detailed detection matrix and export/copy tools.
- Targets: guarded rule configuration and reload controls.
- Settings: static IP, exit binding, startup, recovery, and advanced paths.

## Implementation Order

1. Emergency network recovery.
2. Block reason explanations and repair guidance.
3. Exit IP binding management.
4. First-run setup wizard.
5. Detection report page.

This order prioritizes safety and trust before adding larger UI flows.

## Testing Strategy

- Add unit tests for reason catalog completeness.
- Add service tests for emergency recovery success and partial failure.
- Add service tests for binding reset and rebind behavior.
- Add renderer or static tests for primary UI wiring.
- Add report-generation tests that verify masked IP output.
- Keep existing network checks injectable so tests do not perform real external calls.

## Risks And Open Questions

1. macOS firewall fallback is currently unsupported. The product should explicitly label macOS protection as proxy-based until a privileged helper or Network Extension exists.
2. Free IP intelligence providers can fail or misclassify. v0.2 should explain this clearly rather than pretending the risk score is authoritative.
3. Emergency recovery may require administrator permissions for hosts and firewall cleanup on Windows. The UI should distinguish permission failure from logic failure.
4. The app currently has visible Chinese encoding issues in some files or terminal views. Product trust will suffer if shipped text appears corrupted, so copy normalization should be included before release.
5. `STRICT_VALIDATE` exists in code but is not clearly exposed as a product mode. We should decide whether to expose modes in v0.2 or keep that for a later version.

## Success Metrics

- A new user can enable protection without editing config files.
- A blocked user can identify the cause and next action within 10 seconds.
- A user can restore system network settings with one action.
- Support/debug reports can be shared without exposing full IP addresses.
- Tests cover every supported block reason and recovery action.
